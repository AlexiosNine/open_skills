# 为什么同步 HTTP 调用会导致死锁？

## 核心问题

**FastAPI 异步路由 + 同步阻塞调用 = 事件循环阻塞 = 死锁**

## 详细解释

### 1. FastAPI 的异步模型

FastAPI 使用 **异步路由（async def）** 和 **事件循环（Event Loop）**：

```python
@app.post("/agent/chat")
async def chat(request: AgentRequest):  # ← 异步路由
    # 这个函数运行在事件循环中
    response = agent.chat(request)      # ← 同步方法！
    return response
```

**关键点：**
- `async def` 路由运行在**单线程事件循环**中
- 事件循环一次只能处理一个任务
- 如果某个任务阻塞，整个事件循环都会被阻塞

### 2. 同步阻塞调用

```python
class Agent:
    def chat(self, request):  # ← 同步方法
        # ...
        tool_result = tool_manager.invoke_tool(...)  # ← 内部使用同步 HTTP
        
class ToolManager:
    def invoke_tool(self, ...):
        # 同步 HTTP 调用（阻塞）
        response = requests.post(url, ...)  # ← 阻塞调用！
        return response.json()
```

**关键点：**
- `requests.post()` 是**同步阻塞**的
- 它会**等待** HTTP 响应，期间**不会释放**线程
- 在异步环境中，这会**阻塞事件循环**

    ↓
    `invoke_tool()` 使用 `requests.post()` 同步调用 Skill Host

    ↓
    事件循环无法处理其他请求（包括 Skill Host 的响应）
    ↓

### 4. 为什么是死锁？

**关键点：Agent 和 Skill Host 在同一个 FastAPI 应用中！**

```
┌─────────────────────────────────────────────────┐
│         FastAPI 应用（单进程、单事件循环）        │
│                                                 │
│  ┌──────────────────────────────────────────┐  │
│  │  POST /agent/chat (异步路由)             │  │
│  │  ↓                                       │  │
│  │  agent.chat() [同步方法]                 │  │
│  │  ↓                                       │  │
│  │  requests.post() [阻塞调用]              │  │
│  │  ⚠️ 阻塞事件循环！                       │  │
│  └──────────────────────────────────────────┘  │
│                                                 │
│  ┌──────────────────────────────────────────┐  │
│  │  POST /skills/echo:invoke (异步路由)     │  │
│  │  ❌ 无法处理！事件循环被阻塞              │  │
│  └──────────────────────────────────────────┘  │
│                                                 │
└─────────────────────────────────────────────────┘
```

**死锁链条：**

1. **Agent 路由** (`/agent/chat`) 开始处理请求
2. **Agent.chat()** 调用同步 HTTP 请求
3. **事件循环被阻塞**，无法处理新请求
4. **Skill Host 路由** (`/skills/echo:invoke`) 无法响应（事件循环被占用）
5. **Agent 等待响应**，但 Skill Host 无法处理
6. **死锁！** 🔒

### 5. 为什么直接调用 Runner 可以解决？

```python
# 之前：通过 HTTP（会死锁）
response = requests.post("http://127.0.0.1:8000/skills/echo:invoke", ...)

# 现在：直接调用（不会死锁）
runner = factory.get_runner(manifest)
result = runner.invoke(...)
```

**区别：**

- **HTTP 调用**：需要事件循环处理网络 I/O，但事件循环被阻塞
- **直接调用**：不经过网络，直接执行，不阻塞事件循环

### 6. 可视化对比

#### ❌ 之前（死锁）

```
事件循环线程
├─ 处理 /agent/chat 请求
│  └─ agent.chat() [同步]
│     └─ requests.post() [阻塞！]
│        ⏸️ 等待 HTTP 响应...
│           ⚠️ 事件循环被阻塞
│
└─ ❌ 无法处理 /skills/echo:invoke（事件循环被占用）
```

#### ✅ 现在（正常）

```
事件循环线程
├─ 处理 /agent/chat 请求
│  └─ agent.chat() [同步]
│     └─ runner.invoke() [直接调用，不阻塞]
│        ✅ 立即返回结果
│
└─ ✅ 可以处理其他请求（事件循环未被阻塞）
```

## 总结

**死锁的根本原因：**

1. **FastAPI 使用单线程事件循环**
2. **同步阻塞调用阻塞了事件循环**
3. **Agent 和 Skill Host 在同一进程中**
4. **Agent 等待 Skill Host，但 Skill Host 无法响应（事件循环被阻塞）**

**解决方案：**

- ✅ **直接调用 Runner**（当前方案）
- ✅ **使用异步 HTTP 客户端**（如果分离部署）
- ✅ **使用线程池**（在异步路由中运行同步代码）
